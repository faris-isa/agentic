# GitHub Issue Triage

Triage issues in the current repo using a label-based state machine. Use `gh` for all GitHub operations.

## AI Disclaimer

Every comment or issue posted to GitHub during triage **must** include:

```
> *This was generated by AI during triage.*
```

## Labels

| Label             | Type     | Description                              |
| -----------------| -------- | ---------------------------------------- |
| `bug`             | Category | Something is broken                      |
| `enhancement`     | Category | New feature or improvement               |
| `needs-triage`    | State    | Maintainer needs to evaluate this issue  |
| `needs-info`      | State    | Waiting on reporter for more information |
| `ready-for-agent` | State    | Fully specified, ready for AFK agent     |
| `ready-for-human` | State    | Requires human implementation            |
| `wontfix`         | State    | Will not be actioned                     |

Every issue should have exactly **one** state label and **one** category label.

## State Machine

| Current State  | Can transition to | What happens                                         |
| ---------------| ----------------- | --------------------------------------------------- |
| `unlabeled`    | `needs-triage`    | Issue needs maintainer evaluation                  |
| `unlabeled`    | `ready-for-agent` | Issue is already well-specified                    |
| `unlabeled`    | `ready-for-human` | Requires human implementation                      |
| `unlabeled`    | `wontfix`         | Spam/duplicate/out of scope                         |
| `needs-triage` | `needs-info`      | Issue is underspecified                             |
| `needs-triage` | `ready-for-agent` | Grilling complete, agent-suitable                  |
| `needs-triage` | `ready-for-human` | Grilling complete, needs human                     |
| `needs-triage` | `wontfix`         | Maintainer decides not to action                 |
| `needs-info`   | `needs-triage`    | Reporter has replied                               |

## Workflow: Show What Needs Attention

Query GitHub and present grouped by:

1. **Unlabeled issues** — new, never triaged
2. **`needs-triage` issues** — awaiting evaluation
3. **`needs-info` issues with new activity** — reporter replied

Show oldest first. For each: number, title, age, one-line summary.

## Workflow: Triage a Specific Issue

### Step 1: Gather context
- Read full issue: body, comments, labels, who reported
- Explore codebase for relevant code paths
- Check `.out-of-scope/` for prior rejections

### Step 2: Present recommendation
- **Category**: bug or enhancement, with reasoning
- **State**: where it should go, with reasoning
- Surface any similar out-of-scope rejections

### Step 3: Bug reproduction (bugs only)
- Attempt to reproduce before grilling
- Report findings to maintainer

### Step 4: Grilling session (if needed)
- Interview to build complete specification

### Step 5: Apply outcome
- **ready-for-agent**: write agent brief comment, apply label
- **ready-for-human**: write summary comment, apply label
- **needs-info**: post triage notes with questions
- **wontfix**: close with explanation

## Quick State Override

When maintainer explicitly says "move #42 to ready-for-agent", trust their judgment and apply directly.
